## 📋 BAG_E (v17.19–20) — Детали фикса ✅

**ПРОБЛЕМА:**
`is_journalist_phrase` возвращала `Journalist=True` для фраз типа
"Да, ну давайте еще раз." — Исаев соглашается повторить, но `\bдавайте\b`
в weak_journalist_markers триггерил ложно.

**ROOT CAUSE (5 Whys):**
1. Почему реплики Исаева попали к Журналисту? → pyannote слил 5 сегментов (паузы 0.1–0.3s) в SPEAKER_01
2. Почему split не разделил? → все 5 предложений нейтральные → `neutral → original_speaker = Журналист`
3. Почему предложение [2] не сигнализировало смену? → `\bдавайте\b` давал `Journalist=True` вместо `False`
4. Почему `давайте` = журналист? → паттерн слишком широкий, не различает "давайте вы" и "давайте я/мы/еще"
5. ROOT CAUSE: `\bдавайте\b` без контекста — false positive для фраз эксперта-инициатора

**FIX v17.19:** Добавлен антипаттерн в `is_journalist_phrase`:
```python
r'\bдавайте\s+(я|мы|еще|ещё)\b'
Файл: scripts/corrections/boundary_fixer.py

micro-fragment: conf < -0.25 AND ≤5 слов

merged-диалог: conf < -0.25 AND ≥3 предложений (ловит 00:14:21)
Файл: scripts/export/txt_export.py
FIX: confidence теперь корректно переносится в all_segments dict.

## 📋 BAG_D_v3 (v17.18) — Детали фикса ✅

**ПРОБЛЕМА:**
Таймкод `00:23:37` стоял внутри предложения («...броня до 100 мм, лоб, который был практически  00:23:37  непоражаем...») после успешного real-теста v17.17.

**ROOT CAUSE (5 Whys):**
1. Почему ts внутри предложения? → FALLBACK inject в блоке `00:23:07–00:23:51` (44s) попал на `word#46/71` — середину последнего предложения
2. Почему блок только 44s? → `merge_replicas` остановился: пауза между `00:23:51` и `00:23:53` = **2.6s > 2.0s** (порог для не-Журналиста) без similarity → STOP
3. Почему не склеилось позже? → `auto_merge_adjacent_same_speaker` склеил эти блоки ПОСЛЕ `insert_intermediate_timestamps` — ts уже был вставлен в маленький блок
4. Почему повторный inject в этап 9 не исправил? → В тексте уже был ts `00:23:37`, хвост = `00:25:43 - 00:25:31` = 12s ≤ 45s → SKIP. Повторный проход пропустил весь блок.
5. ROOT CAUSE: перед повторным вызовом `insert_intermediate_timestamps` после `auto_merge` не очищались уже вставленные ts из текстов сегментов

**FIX v17.18:**
В `transcribe.py`, этап 9, перед повторным `insert_intermediate_timestamps`:
```python
# Очищаем ts из текстов перед повторным inject
for seg in segments_merged:
    seg['text'] = re.sub(r'\s*\b\d{2}:\d{2}:\d{2}\b\s*', ' ', seg['text']).strip()
Файл: scripts/transcribe.py

Как найти быстрее:
Сразу после появления бага — запустить grep "auto_merge\|AFTER AUTO-MERGE" debug.log и проверить, есть ли там строки ── Сегмент с повторным inject. Если строк нет — повторный inject был пропущен через SKIP. Дальше проверить tail в логе insert_intermediate_timestamps для этого блока.

## 📋 BAG_B (v17.13) — Детали фикса ✅

**ПРОБЛЕМА:**
`clean_loops` удалял рефраз спикера — фразу «Искры, отцом которого» как дубль
«искры, отцом которой» (sim=0.90), хотя спикер осознанно сменил род («которого/которой»).

**ROOT CAUSE (5 Whys):**
1. Почему «Искры» пропало? → `clean_loops` удалил первое вхождение как дубль (sim=0.90 ≥ 0.75)
2. Почему sim высокое? → N-граммы «Искры, отцом которого» и «искры, отцом которой» отличаются только окончанием одного слова
3. Почему порог 0.75 не защитил? → Порог настроен на обнаружение петель Whisper, но рефраз спикера также даёт высокий sim
4. Почему рефраз не отличался от петли? → Не было проверки контекста после совпадающей N-граммы
5. ROOT CAUSE: `clean_loops` не проверял, что идёт ПОСЛЕ совпавших N-грамм — рефраз всегда имеет разное продолжение

**FIX v17.13:**
LOOKAHEAD guard в `clean_loops()`: перед удалением сравниваем 3 слова после
anchor и 3 слова после candidate (sim порог 0.7).
Если `after_anchor` ≠ `after_cand` (sim < 0.7) → это рефраз → KEEP.
Если `after_anchor` ≈ `after_cand` (sim ≥ 0.7) → это петля → DELETE.

Файл: `scripts/merge/replica_merger.py`

## 📋 BAG_F (v17.11) — Детали фикса ✅

**ПРОБЛЕМА:**
TXT NW_03: инжектированный timestamp `00:16:26` стоял **после** `00:15:59`
(начало следующего сегмента) → инверсия хронологии в финальном тексте.

**ROOT CAUSE (5 Whys):**
1. Почему 00:16:26 > 00:15:59? → `_get_real_time_for_word()` вернула время за пределами seg.end
2. Почему lookup вышел за пределы? → scale=2.287: `total_pre_words=199 / words_post=87`
3. Почему scale=2.287? → `sub_segments` у дочернего сегмента унаследованы от родителя целиком (199 слов), хотя после split осталось только 87 слов
4. Почему sub_segments не пересчитаны после split? → `split_mixed_speaker_segments()` копирует `sub_segments` от родителя, не разбивая их пропорционально
5. ROOT CAUSE: `insert_intermediate_timestamps()` не проверяла валидность scale перед использованием sub_segments

**FIX v17.11:**
Guard в `insert_intermediate_timestamps()` перед построением debug-блока:
```python
_SCALE_ANOMALY_THRESHOLD = 1.8
if has_real_data and words_total > 0:
    _scale = total_pre_words / words_total
    if _scale > _SCALE_ANOMALY_THRESHOLD:
        sub_segments = []
        total_pre_words = 0
        has_real_data = False
        # → fallback ESTIMATED (линейная интерполяция строго внутри seg границ)
```
Файл: `scripts/corrections/timestamp_fixer.py`

Симуляция: `tests/simulations/sim_bug30_timestamp_scale.py`

N-грамма "был. И вот" сравнивалась с якорем "было. и, в" (sim=0.80 ≥ 0.75)

**ROOT CAUSE (5 Whys):**
1. Почему удалилось "был."? → N-грамма "был. И вот" нашла sim ≥ 0.75 с предыдущей
2. Почему sim высокое? → Обе N-граммы состоят только из грамматических слов (форма "быть" + союзы + частицы) — они морфологически похожи без семантики
3. Почему порог 0.75 не защитил? → Порог настроен на семантические повторы, не учитывает грамматические слова
4. Почему в seen[] попал бессодержательный якорь? → Не было фильтра по значимости N-граммы перед добавлением в seen[]
5. ROOT CAUSE: отсутствие проверки семантической значимости N-граммы перед fuzzy matching

**FIX v17.9:**
- Добавлен `RUSSIAN_STOP_WORDS` (104 слова)
- Добавлена `_count_meaningful(phrase)` — считает слова не из стоп-листа
- В `clean_loops()`: N-граммы с `_count_meaningful < MIN_MEANINGFUL_WORDS=2` сохраняются в output, но в `seen[]` не добавляются
